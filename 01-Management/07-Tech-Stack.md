## 二、核心技术

### 1. Markdown

**用途**: 所有文档的载体  
**版本**: CommonMark + GitHub Flavored Markdown  
**文件扩展名**: `.md`

**使用规范**:
- 使用UTF-8编码
- 使用Unix换行符（LF）
- 最大文件大小：建议不超过100KB（约5000行）

**支持的语法**:
- 标题（H1-H6）
- 列表（有序/无序）
- 表格
- 代码块
- 链接和图片
- 引用
- 任务列表（`- [ ]` / `- [x]`）

**示例**:
```markdown
# 一级标题
## 二级标题

- 列表项1
- 列表项2

| 列1 | 列2 |
|-----|-----|
| 内容1 | 内容2 |

[链接文本](文件路径.md)
```

---

### 2. Git

**用途**: 版本控制  
**版本**: 2.x  
**托管平台**: GitHub

**仓库地址**: https://github.com/ynych/redream

**分支策略**:
- 单分支模式：`main` 分支
- 无功能分支，所有更改直接提交到main
- 适合单人/少人协作项目

**提交规范**:
```
[模块] 简要描述

详细说明（可选）

- 变更点1
- 变更点2
```

**示例**:
```
[Management] 汉化并优化 01-Management 目录

- 新增 00-Project-Navigation.md（中文）
- 删除 00-README.md（英文版）
- 删除 05-Project-Backup.md（过时备份）
- 重命名所有文件为中文名
```

**常用命令**:
```bash
# 查看状态
git status

# 添加文件
git add -A

# 提交
git commit -m "[模块] 描述"

# 推送
git push origin main

# 查看历史
git log --oneline -10

# 拉取更新
git pull origin main
```

### 3. Shell 脚本

**用途**: 自动化任务  
**位置**: `99-Scripts/`  
**语言**: Bash

**现有脚本**:
- 暂无（可添加日常状态报告、文件检查等）

**建议脚本**:
```bash
# 项目状态报告
#!/bin/bash
echo "=== 项目状态报告 ==="
echo "日期: $(date)"
echo "Git提交数: $(git rev-list --count HEAD)"
echo "文件总数: $(find . -name '*.md' | wc -l)"
echo "最近提交: $(git log -1 --pretty=format:'%h %s')"
```

## 四、文件组织规范

### 目录命名
- 使用英文
- 使用编号前缀：`01-`, `02-`, `03-`...
- 使用驼峰命名或首字母大写：`Management`, `Phase2-Outline`

**正确示例**:
```
01-Management/
02-Planning/
03-Design/
```

**错误示例**:
```
01-管理/          # 中文目录名
management/       # 无前缀
01_management/    # 下划线分隔
```

### 文件命名
- 使用英文
- 使用kebab-case（短横线分隔）或CamelCase
- 编号前缀用于排序：`00-`, `01-`, `02-`...

**正确示例**:
```
00-Project-Navigation.md
foreshadowing-table-v2.md
Outline_Confirmation_Checklist.md
```

**错误示例**:
```
项目导航.md        # 中文文件名
project_navigation.md  # 下划线
00 Project Navigation.md  # 空格
```

### 内容语言
- **所有内容使用简体中文**
- 代码/脚本使用英文
- 专有名词保留（GitHub, Markdown, Agent名称等）

## 五、Git工作流

详见: `08-Git-Workflow.md`

**简化版**:
```
编辑文件 → git add → git commit → git push
```

**详细步骤**:
1. 在OpenClaw中编辑文件
2. 保存到本地工作区
3. `git add -A` 添加所有变更
4. `git commit -m "[模块] 描述"` 提交
5. `git push origin main` 推送到GitHub

## 六、备份策略

### 自动备份
- GitHub自动保存所有提交历史
- 每次提交都是一次备份点

### 本地备份
- 建议定期clone仓库到本地备份
- 或者使用GitHub的导出功能

### 灾难恢复
- 如果GitHub不可用：使用本地clone恢复
- 如果本地丢失：从GitHub重新clone

## 七、性能考虑

### 文件大小
- 单个Markdown文件建议不超过100KB
- 大文件可拆分为多个小文件

### 仓库大小
- 当前：约5MB
- 预计最终：20-50MB（含28回×3版本）
- 如果超过100MB：考虑使用Git LFS

### 响应速度
- GitHub访问：依赖网络
- 本地编辑：即时响应
- AI协作：受token限制

---

## 八、安全考虑

### 敏感信息
- **禁止在文档中包含密码、API Key等敏感信息**
- 使用环境变量或单独配置文件（不提交到Git）

### 访问控制
- GitHub仓库：私有（仅授权用户可见）
- 本地文件：操作系统权限控制

### 备份安全
- GitHub有企业级安全保护
- 本地备份需自行保护
