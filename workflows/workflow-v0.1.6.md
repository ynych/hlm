## 一、文档结构优化（解决上下文超长）

```
deliverables/v0.1.6/fulltext/
├── fulltext-v0.1.6-part81-87.md    # 第81-87回
├── fulltext-v0.1.6-part88-94.md    # 第88-94回
```

- 版本信息

## 二、v0.1.6工作流程（6阶段）

### Phase 1: 4+1专家选择（Day 0）

```
□ 1.1 从11位专家库随机选4位
□ 1.2 分析4位专家的领域覆盖
□ 1.3 生成+1专家（填补最大空白）
□ 1.4 将+1专家加入专家库（变为12位）
□ 1.5 发送评审任务书给5位专家
```

### Phase 2: 4+1专家评审（Day 1-2）

**评审方式**: 分文档并行评审

```
专家A → part81-87.md
专家B → part88-94.md
专家C → part95-101.md
专家D → part102-108.md
+1专家 → 全文索引+重点回目
```

**Day 1**: 并行评审
**Day 2**: 交叉验证 + 冲突仲裁

### Phase 3: 问题汇总与方案设计（Day 3）

```
□ 3.1 汇总5份专家评审报告
□ 3.2 分类问题（P0/P1/P2）
□ 3.3 关联问题到具体文档（part81-87等）
□ 3.4 设计优化方案
    - 方案A: 全面冲刺（所有P0+P1+P2）
    - 方案B: 精准优化（高价值P0+P1）
    - 方案C: 保守迭代（仅P0）
□ 3.5 用户确认方案
```

### Phase 4: 分文档修复执行（Day 4-7）

**按文档分批修复**:

```
Day 4: part81-87.md 修复
Day 5: part88-94.md 修复
Day 6: part95-101.md 修复
Day 7: part102-108.md 修复
```

**每个文档流程**:
1. 读取文档（~6万字，上下文安全）
2. 定位修复位置
3. 执行修复
4. 自查
5. 更新索引

### Phase 5: 整合与验证（Day 8）

```
□ 5.1 合并4份文档为完整全文
□ 5.2 跨文档一致性检查
□ 5.3 生成完整评审汇总
□ 5.4 专家快速确认（可选）
```

### Phase 6: 发布（Day 9）

```
□ 6.1 更新版本号v0.1.6
□ 6.2 GitHub推送
□ 6.3 生成发布说明
□ 6.4 归档文档
```
