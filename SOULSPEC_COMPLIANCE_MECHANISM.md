# Redream 项目专家 SOULSPEC 合规与能力培养机制

**版本**: v1.0  
**制定日期**: 2026-04-06  
**适用范围**: 《癸酉续红》项目所有专家  
**核心目标**: 确保每位专家符合SOULSPEC标准，具备多Agent协同和自进化能力

---

## 一、SOULSPEC 合规认证机制

### 1.1 三阶段认证流程

```
入职认证 → 任务认证 → 晋升认证
    ↓           ↓           ↓
  基础合规    实战验证    能力升级
```

#### 阶段一：入职认证（必须）

**时间**: 专家首次参与项目前  
**负责**: Echo（项目统筹）

**认证清单**:

| 检查项 | 要求 | 验证方式 | 不达标处理 |
|--------|------|----------|------------|
| 档案完整 | 包含8个必需字段 | 档案审核 | 限期3天补全 |
| SOULSPEC声明 | 明确签署v1.0合规声明 | 文档检查 | 不予入职 |
| 协作接口 | 定义输入/输出/依赖 | 接口文档 | 限期2天定义 |
| 禁令认知 | 知晓5大禁止行为 | 口头/书面测试 | 培训后重测 |

**必需档案字段**:
```yaml
agent_id: [唯一ID]
name: [中文名]
role: [角色定位]
soulspec_version: v1.0
compliance_date: [签署日期]
interfaces:
  input: [接受任务类型]
  output: [交付成果类型]
  dependencies: [依赖的其他Agent]
main_task:
  scope: [负责范围]
  deliverable: [交付物标准]
  deadline: [截止日期]
evolution_log: [进化日志链接]
```

#### 阶段二：任务认证（每次任务）

**时间**: 每次分配任务前  
**负责**: 分卷主编

**任务合规检查**:

```markdown
## 任务合规检查单

- [ ] 任务范围是否在expert.scope内？
- [ ] 是否有明确的deliverable标准？
- [ ] deadline是否合理（至少预留20%缓冲）？
- [ ] 是否知晓依赖的Agent并确认协作接口？
- [ ] 是否理解禁令（版本号/擅自决策等）？

检查人: [分卷主编签名]
日期: [日期]
```

#### 阶段三：晋升认证（升级时）

**时间**: 从候补→正式，或正式→核心  
**负责**: 编委会

**晋升标准**:

| 级别 | 任务完成率 | 质量评分 | 协作评价 | 自进化记录 |
|------|------------|----------|----------|------------|
| 候补→正式 | ≥80% | ≥75分 | 无投诉 | ≥3条 |
| 正式→核心 | ≥90% | ≥85分 | 优秀评价 | ≥10条 |
| 核心→主编 | ≥95% | ≥90分 | 领导力认可 | ≥20条 |

### 1.2 合规检查工具

#### 自动化检查脚本

```bash
#!/bin/bash
# soulspec-check.sh

EXPERT_ID=$1

echo "=== SOULSPEC Compliance Check for $EXPERT_ID ==="

# 检查档案存在
if [ ! -f "experts/${EXPERT_ID}-profile.md" ]; then
    echo "❌ FAIL: Profile not found"
    exit 1
fi

# 检查必需字段
required_fields=("agent_id" "name" "role" "soulspec_version" "interfaces" "main_task")
for field in "${required_fields[@]}"; do
    if ! grep -q "$field" "experts/${EXPERT_ID}-profile.md"; then
        echo "❌ FAIL: Missing field '$field'"
        exit 1
    fi
done

# 检查SOULSPEC版本
if ! grep -q "soulspec_version: v1.0" "experts/${EXPERT_ID}-profile.md"; then
    echo "❌ FAIL: SOULSPEC version not v1.0"
    exit 1
fi

# 检查进化日志
if [ ! -f "experts/${EXPERT_ID}-evolution.md" ]; then
    echo "⚠️ WARN: Evolution log not found"
fi

echo "✅ PASS: Basic compliance verified"
```

## 二、多Agent协同机制

### 2.1 协同架构

```
项目层协同
├── 编委会（每周三会议）
│   ├── 总编（用户）- 最终决策
│   ├── 主编（Echo）- 统筹协调
│   ├── 质量总监（ARC-001）- 质量把关
│   └── 4位分卷主编 - 分卷负责
│
├── 任务层协同（每次任务）
│   ├── 任务发布 - 分卷主编
│   ├── 接口确认 - 执行专家
│   ├── 依赖协调 - Echo
│   └── 成果交付 - 执行专家
│
└── 评审层协同（每次评审）
    ├── 初审 - 同卷专家
    ├── 交叉审 - 跨卷专家
    └── 终审 - 质量总监
```

### 2.2 协同接口规范

#### 输入接口（每项任务必须包含）

```yaml
task_input:
  task_id: [唯一ID]
  task_type: [评审/写作/修复]
  scope: [具体范围]
  
  # 上下文
  context:
    previous_version: [前序版本]
    related_experts: [相关专家ID列表]
    blocking_issues: [阻塞问题]
  
  # 交付标准
  deliverable:
    format: [交付格式]
    criteria: [验收标准]
    examples: [参考样例]
  
  # 约束条件
  constraints:
    deadline: [截止日期]
    word_count: [字数范围]
    style_guide: [风格指南链接]
    banned_words: [禁用词列表]
```

#### 输出接口（每项交付必须包含）

```yaml
task_output:
  task_id: [对应输入ID]
  expert_id: [执行专家ID]
  submit_date: [提交日期]
  
  # 交付物
  deliverable:
    file_path: [文件路径]
    checksum: [校验和]
  
  # 元数据
  metadata:
    actual_word_count: [实际字数]
    time_spent: [耗时]
    tools_used: [使用的工具]
  
  # 自检报告
  self_check:
    soulspec_compliant: [是/否]
    quality_score: [自评分数]
    known_issues: [已知问题]
  
  # 协作记录
  collaboration:
    consulted_experts: [咨询过的专家]
    feedback_received: [收到的反馈]
    changes_made: [做出的修改]
```

### 2.3 协同冲突解决机制

#### 冲突类型与升级路径

```
技术分歧（如写法选择）
├── 当事专家协商（1天内）
├── 分卷主编裁决（1天内）
└── 编委会投票（2天内）

创意分歧（如意象选择）
├── 提供多方案给用户
├── 用户选择
└── 执行

版本分歧（如版本号变更）
└── 直接提交用户裁决（禁止Agent决定）
```

## 三、自进化能力培养机制

### 3.1 自进化三要素

```
错误记录 → 能力升级 → 知识传承
    ↓           ↓           ↓
  发现不足    技能提升    团队共享
```

### 3.2 错误记录机制

#### 强制记录场景

专家必须在以下场景记录错误：

1. **任务失败**（未按时交付或质量不达标）
2. **评审未通过**（被其他专家打回）
3. **用户反馈负面**（用户指出问题）
4. **违反SOULSPEC**（如擅自升级版本号）
5. **协作冲突**（与其他专家产生分歧）

#### 错误记录模板

```markdown
## 错误记录 #[ERROR-XXX]

**专家ID**: [ID]
**日期**: [日期]
**错误类型**: [技术/协作/规范/沟通]
**严重级别**: [P0致命/P1严重/P2一般]

### 错误描述
[具体发生了什么]

### 影响范围
[影响了什么/谁]

### 根因分析
为什么会发生？
- [原因1]
- [原因2]

### 修复措施
已采取的行动：
- [措施1]
- [措施2]

### 预防措施
如何避免再次发生：
- [措施1]
- [措施2]

### 经验教训
学到了什么：
[总结]
```

#### 错误分类与处理

| 级别 | 定义 | 处理 | 记录要求 |
|------|------|------|----------|
| P0 | 致命错误（如擅自改版本号） | 立即上报编委会 | 24小时内记录 |
| P1 | 严重错误（如任务失败） | 分卷主编介入 | 48小时内记录 |
| P2 | 一般错误（如格式问题） | 自行修复 | 1周内记录 |

## 四、监督与考核机制

### 4.1 三级监督体系

```
自我监督（每日）
├── 每日同步报告
├── 任务进度自检
└── 错误及时记录

同级监督（每周）
├── 同卷专家互审
├── 协作反馈收集
└── 周会问题提出

上级监督（每月）
├── 分卷主编评估
├── 质量总监抽检
└── 编委会月度评审
```

### 4.2 考核指标

#### 量化指标（权重60%）

| 指标 | 权重 | 优秀 | 良好 | 合格 | 不合格 |
|------|------|------|------|------|--------|
| 任务完成率 | 20% | ≥95% | ≥90% | ≥80% | <80% |
| 质量评分 | 20% | ≥90 | ≥85 | ≥75 | <75 |
| 按时交付率 | 10% | ≥95% | ≥90% | ≥80% | <80% |
| 协作评价 | 10% | 优秀 | 良好 | 合格 | 差 |

#### 质化指标（权重40%）

| 指标 | 权重 | 评估方式 |
|------|------|----------|
| SOULSPEC合规 | 15% | 合规检查工具 |
| 自进化活跃度 | 15% | 进化日志数量和质量 |
| 知识贡献 | 10% | 知识分享和传承 |

### 4.3 奖惩机制

#### 奖励

| 成就 | 奖励 |
|------|------|
| 连续3个月优秀 | 晋升核心专家 |
| 提出重大改进 | 编委会表彰 |
| 成功带新人 | 额外资源分配 |
| 月度最佳 | 公开表扬 |

#### 惩罚

| 违规 | 惩罚 |
|------|------|
| 违反SOULSPEC（如擅自改版本号） | 警告+强制培训 |
| 连续2次任务失败 | 降级为候补 |
| 连续3次延期 | 降级为候补 |
| 重大协作冲突 | 调岗观察 |
| 泄露项目资料 | 立即除名 |

### 4.4 申诉机制

专家如对考核结果有异议，可提出申诉：

```
申诉流程:
1. 收到考核结果3天内提出申诉
2. 提交申诉材料（证据+理由）
3. 编委会组成申诉小组（3人，不含当事人）
4. 5天内给出申诉结果
5. 结果为最终裁决
```
